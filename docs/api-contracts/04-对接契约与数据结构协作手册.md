# 对接契约与数据结构协作手册

> 本文件专门放三人协作边界、接口契约、事件契约、状态机、核心字段和数据库结构要点。个人任务文档不展开接口细节，避免重复和漂移。

## 0\. 使用方式

任何跨人协作都先更新本文件或 `docs/api\-contracts/`：

- FE 根据本文件接页面。

- BE\-A 根据本文件实现审核智能业务引擎。

- BE\-B 根据本文件提供平台支撑与数据资产服务。

规则：

- 字段名一旦前后端确认，不允许单方修改。

- 状态值一旦确认，不允许单方新增。

- 数据库字段变更必须同步说明影响的接口和页面。

- 接口错误码必须能指导前端展示。

- 事件字段变更必须说明生产方、消费方和幂等键。

## 2\. 解耦规则

### 2\.1 数据集与模板

BE\-A 不直接维护数据集原始文件和模板版本表。

BE\-A 需要数据时调用 BE\-B：

```Plaintext
getDatasetItemSnapshot(itemId)
findClaimableItem(taskId, labelerId)
getTemplateSchema(templateVersionId)
validateAnswer(schemaVersionId, answerJson)
```

BE\-B 不直接修改 assignment、submission、review 状态。

领取一致性采用 MVP 模块化单体方案：

```Plaintext
BE-A 编排领取用例
BE-A 开启本地事务
BE-A 调用 BE-B reserveClaimableItem(taskId,labelerId)
BE-A 创建 assignment
同一事务提交或回滚
```

未来如果拆成独立服务，改为：

```Plaintext
BE-B reserveClaimableItem -> 返回 reservationId
BE-A 创建 assignment
BE-A confirm reservation
失败时 reservation 过期释放
```

### 2\.2 审核状态

submission 状态、review 状态、isGolden 由 BE\-A 决定。

课程验收口径：

```Plaintext
课程验收阶段按单级 Reviewer 终审交付。
reviewLevel 仅作为扩展字段保留。
MVP 不启用初审/复审/终审多级流转。
P1 再扩展 reviewLevel=1/2/3。
```

BE\-B 可以读取这些状态做：

- 贡献统计。

- 奖励结算。

- 导出筛选。

- 看板聚合。

### 2\.3 任务分发策略

MVP 只实现：

```Plaintext
先到先得 + 任务级配额 + 用户领取锁定
```

不实现：

```Plaintext
Labeler 指派。
配额抢单之外的复杂调度。
按组织/团队定向分发。
```

说明：

```Plaintext
Reviewer 批量分配只表示审核员分配，不是标注任务指派。
Labeler 任务领取始终走 claim 流程和 Redis lock。
```

### 2\.4 AI 审核

AI 审核由 BE\-A 闭环：

```Plaintext
agent_run 创建
LLM Provider 选择
LlmGateway 调用
AI 结果写入
submission 推进到 PENDING_FINAL
AI 失败兜底时 aiReview.status=MANUAL_REQUIRED，submission.status 仍为 PENDING_FINAL
```

BE\-B 只提供：

- RateLimitService。

- AsyncExecutor。

- Redis 基础能力。

System Principal 边界：

```Plaintext
BE-B 创建和维护 system_ai_agent 用户。
BE-A 读取 system_ai_agent profile。
BE-A 在 AI 审核时构造 SystemActorContext。
BE-A 调用 AuditAppender 写 actorType=SYSTEM_AGENT 的审计。
```

### 2\.5 奖励统计

BE\-A 不直接写奖励流水。

BE\-A 在业务状态确定后发布事件：

```Plaintext
SubmissionApproved
GoldenSelected
RewardReversed
```

BE\-B 消费事件：

- 生成 reward\_ledger。

- 更新 labeler\_contribution\_stats。

- 更新 dataset\_items\.approvedCount。

- 写奖励审计。

### 2\.6 导出

BE\-B 负责导出文件生成和下载历史。

导出数据范围由 BE\-A 状态决定：

```Plaintext
submission.status = APPROVED
submission.isGolden = true
```

BE\-B 不直接拼装 submission 表数据，必须调用 BE\-A 快照查询：

```Plaintext
queryExportableGoldenSubmissions(taskId, pageRequest)
```

返回内容：

```Plaintext
submissionId
datasetItemId
itemSnapshot
answerJson
aiReviewSnapshot
auditRefs
```

### 2\.7 Auth Propagation

Auth/RBAC 由 BE\-B 提供，BE\-A 复用统一认证上下文。

MVP 采用模块化单体：

```Plaintext
BE-B 签发 JWT
统一 Auth Filter 解析 JWT
统一 CurrentUserContext 暴露 userId/roles/tokenVersion
BE-A 接口从 CurrentUserContext 读取身份
```

角色变更失效机制：

```Plaintext
Admin 修改角色
BE-B 递增 users.tokenVersion
后续请求校验 JWT tokenVersion 与数据库一致
不一致则返回 401001
```

不采用：

```Plaintext
不做第二套 BE-A 登录。
MVP 不做远程 introspection。
MVP 不强依赖 API Gateway 鉴权。
```

### 2\.8 AuditAppender

审计表由 BE\-B 维护结构，A/B 都通过追加接口写审计。

```Plaintext
AuditAppender.append(AuditCommand)
```

`AuditCommand` 必须包含：

```Plaintext
actorType
actorId
bizType
bizId
action
beforeJson
afterJson
traceId
agentRunId optional
```

禁止：

```Plaintext
BE-A 直接操作 audit_logs 表。
BE-B 之外的模块修改审计表结构。
审计日志更新或删除。
```

## 5\. 统一响应与错误码

统一响应：

```JSON
{
  "code": 0,
  "message": "ok",
  "data": {},
  "traceId": "trace-id"
}
```

分页响应：

```JSON
{
  "items": [],
  "page": 1,
  "pageSize": 20,
  "total": 0
}
```

核心错误码：

|code|含义|前端处理|
|---|---|---|
|401001|未登录|跳登录|
|403001|无权限|展示无权限页|
|400101|状态不允许|展示业务错误|
|400102|参数非法|表单字段提示|
|409101|版本冲突|提示刷新或合并|
|409201|领取冲突|提示重新领取|
|409301|Schema 校验失败|定位字段错误|
|429001|LLM 限流|展示等待或重试中|
|500001|系统错误|展示 traceId|

## 6\. 状态机

### 6\.1 TaskStatus

```Plaintext
DRAFT
PUBLISHED
PAUSED
ENDED
```

合法迁移：

```Plaintext
DRAFT -> PUBLISHED
PUBLISHED -> PAUSED
PAUSED -> PUBLISHED
PUBLISHED -> ENDED
PAUSED -> ENDED
```

Owner：BE\-A。

### 6\.2 AssignmentStatus

```Plaintext
CLAIMED
DRAFTING
SUBMITTED
RETURNED
APPROVED
CANCELLED
```

Owner：BE\-A。

合法迁移：

```Plaintext
CLAIMED -> DRAFTING
DRAFTING -> SUBMITTED
CLAIMED -> SUBMITTED
SUBMITTED -> RETURNED
RETURNED -> DRAFTING
RETURNED -> SUBMITTED
SUBMITTED -> APPROVED
CLAIMED -> CANCELLED
DRAFTING -> CANCELLED
RETURNED -> CANCELLED
```

触发方：

```Plaintext
CLAIMED/DRAFTING/SUBMITTED：Labeler 操作触发，BE-A 推进。
RETURNED：Reviewer 打回触发，BE-A 推进。
APPROVED：Reviewer 通过或金标选择后，BE-A 推进。
CANCELLED：任务结束、超时释放或管理员处理触发，BE-A 推进。
```

约束：

```Plaintext
RETURNED 后重新提交会生成新的 submission version。
APPROVED 后 Labeler 不可继续修改。
CANCELLED 后不可提交。
```

### 6\.3 SubmissionStatus

```Plaintext
SUBMITTED
AI_REVIEWING
PENDING_FINAL
APPROVED
REJECTED
SUPERSEDED
```

Owner：BE\-A。

### 6\.4 AiReviewStatus

```Plaintext
PENDING
RUNNING
SUCCESS
FAILED
RATE_LIMITED
MANUAL_REQUIRED
```

Owner：BE\-A。

### 6\.5 ConflictStatus

```Plaintext
NONE
CONSENSUS_REACHED
CONFLICTED
RESOLVED
```

Owner：BE\-A。

## 8\. 事件契约

### 8\.1 SubmissionApproved

Producer：BE\-A。

Consumer：BE\-B。

触发：

```Plaintext
Reviewer approve submission
batch approve submission
```

字段：

```Plaintext
eventId
eventType=SubmissionApproved
taskId
datasetItemId
assignmentId
submissionId
labelerId
reviewerId
approvedAt
```

幂等键：

```Plaintext
eventType + submissionId
```

消费效果：

```Plaintext
生成正向 reward_ledger。
更新 labeler_contribution_stats。
更新 dataset_items.approvedCount。
正向奖励幂等最终以 submissionId 为准。
```

### 8\.2 GoldenSelected

Producer：BE\-A。

Consumer：BE\-B。

触发：

```Plaintext
Reviewer resolve conflict group
```

字段：

```Plaintext
eventId
eventType=GoldenSelected
taskId
datasetItemId
conflictGroupId
goldenSubmissionId
labelerId
reviewerId
resolvedAt
```

幂等键：

```Plaintext
eventType + conflictGroupId
```

消费效果：

```Plaintext
生成正向 reward_ledger。
更新 labeler_contribution_stats。
更新 dataset_items.approvedCount。
如果 goldenSubmissionId 已经产生正向奖励，不重复计奖。
```

### 8\.3 RewardReversed

Producer：BE\-A 或 Admin 操作。

Consumer：BE\-B。

触发：

```Plaintext
approved submission rollback
golden submission rollback
```

字段：

```Plaintext
eventId
eventType=RewardReversed
taskId
submissionId
labelerId
reason
operatorId
createdAt
```

## 13\. 变更流程

任何接口、字段、状态或事件变更必须按顺序：

1. 修改本文件或 `docs/api\-contracts/`。

2. 在团队同步变更影响。

3. 后端改 DTO 和测试。

4. 前端改类型和页面。

5. 联调验证。

禁止：

- 先改代码再口头通知。

- 后端临时改字段名。

- 前端绕过字段类型用 `any`。

- 数据库字段变更不说明历史数据影响。

- 事件新增字段不说明生产方和消费方。
